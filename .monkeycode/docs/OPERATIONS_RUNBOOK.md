# 上线运维制度项

## 1. 发布准入制度
- 发布前必须通过 `npm run lint`、`npm run typecheck`、`npm run regression`、`npm run e2e`。
- 发布前必须确认 `GET /api/health` 返回 200，`GET /api/ready` 返回 200。
- 发布前必须完成一次数据备份，并保存备份文件名、执行时间、执行人。
- 发布前必须确认生产环境数据库连接参数已配置，`ALLOW_FILE_DB_FALLBACK` 保持关闭。

## 2. 变更审批制度
- 生产变更必须记录变更目的、影响范围、执行窗口、回滚方案。
- 涉及数据库、升级包、账号权限、系统配置的变更必须双人复核。
- 高风险变更安排在低峰期执行，并指定观察人和值班人。

## 3. 发布执行制度
- 发布顺序固定为：备份 -> 健康检查 -> 应用发布 -> 就绪检查 -> 冒烟验证。
- 发布时优先保留旧版本包、当前数据库备份、升级前备份。
- 发布完成后必须验证登录、项目查询、资产查询、日志新增、AI 智能巡检、报表导出。

## 4. 备份与恢复制度
- 每日执行数据库备份，备份文件至少保留 7 天。
- 每次发布前必须执行一次手动备份。
- 每月至少进行一次恢复演练，验证导入后当前会话保留、关键业务数据完整。
- 备份目录必须具备可读写权限，并纳入磁盘容量监控。

## 5. 监控与告警制度
- 监控范围至少包括：应用进程、MySQL 连通性、磁盘使用率、内存使用率、备份目录可写性。
- 每 1 分钟采集一次健康状态，每 5 分钟采集一次系统负载。
- `GET /api/ready` 返回非 200 时立即告警。
- AI 智能巡检出现“严重”结果时需要人工复核处理闭环。

## 6. 安全管理制度
- 生产环境必须通过 HTTPS 暴露，并确保登录 Cookie 使用 `Secure`。
- 管理员账号必须使用强密码，禁止共享账号。
- 连续登录失败达到限制后，由值班人员排查来源 IP 与账号行为。
- 升级包必须来自受控来源，发布前校验 `manifest.json` 与文件清单。

## 7. 日志与审计制度
- 操作日志、升级记录、导入导出、重置与备份操作必须保留审计记录。
- 审计日志至少保留最近 500 条，长期归档由外部日志平台承担。
- 审计日志每日归档一次，归档文件按日期分目录保存，至少保留 180 天。
- 生产问题处理完成后，需要补充故障时间线、影响面、修复动作、预防措施。

## 8. 备份与通知清理制度
- 备份文件每周执行一次保留期检查，超过 30 天的本地备份转移到离线归档区。
- 备份目录保留最近 7 天的快速恢复副本，归档区保留最近 12 个月的月度备份。
- 通知记录每日清理一次已读且超过 30 天的数据，清理前导出归档快照。
- AI 智能巡检、审批流转、系统升级相关通知至少保留 90 天。

## 9. 回滚制度
- 每次发布都必须具备明确回滚条件与执行命令。
- 回滚触发条件包括：关键接口失败、登录不可用、数据写入异常、AI 巡检主流程失败。
- 回滚顺序固定为：停止新版本 -> 恢复旧版本代码 -> 恢复数据库备份 -> 执行冒烟验证。

## 10. 值班与响应制度
- 发布窗口内必须有开发、测试、运维至少 1 人在线。
- 严重故障 10 分钟内完成响应，30 分钟内给出初步处置结论。
- 值班交接必须包含当前版本号、已知风险、备份位置、未关闭告警。

## 11. 上线后观察制度
- 发布后 30 分钟内重点观察登录、数据库、系统负载、AI 巡检调度、报表导出。
- 发布后 24 小时内复核关键审计日志、备份状态、通知队列与异常记录。
- 若观察期内出现高频错误，立即进入回滚评估流程。
